# Claude Skills — Technical Deep-Dive Report

## Summary

An 18-page technical report on how Claude Skills work under the hood: runtime architecture, the three-level progressive disclosure model, Skills API endpoints and SDK usage, sandboxing/security model, data privacy caveats, and known failure modes with mitigations. The report was generated by ChatGPT deep research, citing Anthropic's official docs, engineering blogs (sandboxing, code execution with MCP), the 32-page Anthropic skill guide, and community analyses (Scott Spence, Reddit). The two architecture diagram PNGs in `raw/attachments/` ("Agent + Skills + Computer", "Skills and the Context Window") are figures from this report.

## Key Points

### Runtime architecture
- Skills are filesystem directories on Claude's VM; Claude traverses them with OS-level tools (bash, `cat`, Python)
- **Three-level progressive disclosure**: (1) metadata — every installed skill's name + description pre-loaded at session start, ~100 tokens per skill; (2) instructions — `SKILL.md` read into context on trigger match, up to ~5K tokens; (3) resources/code — referenced files read or executed on demand
- Scripts and code never load into model context — only their *outputs* do; unused files impose no token cost

### API and deployment
- Skills run in the code-execution container; API calls need beta headers (code-execution, skills, files-api) and the skill list in the `container` parameter (up to 8 skills per request)
- Built-in skills (`pptx`, `xlsx`, `docx`, `pdf`; ~17 official) use short IDs and date versions; custom skills get generated IDs (`skill_01...`) and epoch-timestamp versions
- Claude Code integration is filesystem-based (drop folders in `.claude/skills/`); claude.ai accepts ZIP uploads (paid plans)
- Generated files come back as `file_id`s fetched via the Files API; long tasks use the `pause_turn` stop reason for chunking

### Security and privacy
- Sandboxing via OS primitives (bubblewrap on Linux, seatbelt on macOS); no direct network — outbound calls go through Anthropic proxies with domain whitelists
- Skills are **not covered by Zero Data Retention** — relevant for compliance planning
- Code execution as privacy tool: sensitive data can flow through scripts without ever entering model context
- Skills should be trusted/reviewed before install — a malicious skill can abuse any tool capability it has within the sandbox

### Failure modes and mitigations
- **Trigger unreliability** is the top community complaint: skills often fail to activate. Mitigations: explicit invocation, forced-evaluation prompting ("for each skill say yes/no then activate"), multiple trigger phrases in descriptions
- Prefer **micro-skills** over monolithic ones (context limits: instructions near 5K tokens crowd the window)
- Each installed skill adds ~100 tokens of always-loaded metadata — uninstall unused skills
